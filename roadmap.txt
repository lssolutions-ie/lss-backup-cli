LSS Backup CLI V2 Roadmap
=========================

Purpose
-------
This roadmap is for building a cleaner, safer, and more maintainable v2 of
LSS Backup CLI while preserving the strongest ideas from v1:

- each backup job should be independently restorable
- the job definition should be durable across reinstall/update
- import/export should remain first-class
- restic and rsync should both be supported
- the product should stay operator-friendly
- nodes should report backup status to a central management server


Guiding Principles
------------------
- One job definition file should be the source of truth.
- Runtime artifacts should be disposable and rebuildable.
- Jobs should remain operationally independent.
- Backup logic should live in shared code, not duplicated per job.
- Config should be treated as data, not executable script.
- Secrets should be stored separately from general config.
- Safety and recoverability matter more than cleverness.
- Reporting must never fail or block a backup job.


Target V2 Shape
---------------
Suggested high-level structure:

- `bin/`
- `jobs/<job-id>/job.toml`
- `jobs/<job-id>/secrets.env`
- `jobs/<job-id>/last_run.json`        <- added: persisted run result
- `jobs/<job-id>/logs/`
- `jobs/<job-id>/run.sh`
- `state/`
- `config.toml`                        <- added: app-level config (reporting)
- `docs/`

Suggested CLI commands:

- `lss-backup init`
- `lss-backup create`
- `lss-backup list`
- `lss-backup show <job-id>`
- `lss-backup run <job-id>`
- `lss-backup validate <job-id>`
- `lss-backup restore <job-id>`
- `lss-backup remove <job-id>`
- `lss-backup import <path>`
- `lss-backup export <job-id>`
- `lss-backup schedule sync`

Milestone M7: Management Server Integration
-------------------------------------------
Goal:
Allow nodes to report backup job results to a central management server
over HTTPS. Reporting is always non-blocking and non-destructive.

Design decisions locked in:
- App-level config lives at `{RootDir}/config.toml`, mode 0600.
- Per-node token stored there; admin creates the token in the dashboard.
- Authorization via `Authorization: Bearer {token}` header (not request body).
- Payload always includes `payload_version: "1"` for forward compatibility.
- 15-second HTTP timeout; failure is logged as a warning in the job log only.
- No retry queue for now. Fire-and-forget.
- Node name is user-configurable (not derived from hostname) because VMs
  get renamed and cloned.

Tasks:
- [ ] Add `AppConfig` struct with a `[reporting]` section:
        enabled     bool
        server_url  string
        node_token  string
        node_name   string
- [ ] Add load/save for `{RootDir}/config.toml` (TOML, created on first run).
- [ ] Add "Configure Reporting" option to the CLI main menu:
        - enable/disable toggle
        - set server URL
        - set node token (masked input, like restic password)
        - set node name
- [ ] Implement `HTTPReporter` in `internal/reporting/`:
        - POST to `{server_url}/api/v1/status`
        - Auth header: `Authorization: Bearer {node_token}`
        - 15s timeout
        - log warning on failure, never return error that surfaces to user
- [ ] Define and lock the JSON payload shape:
        {
          "payload_version": "1",
          "node_name":        string,
          "job_id":           string,
          "job_name":         string,
          "program":          string,
          "status":           "success" | "failure",
          "started_at":       RFC3339 UTC,
          "finished_at":      RFC3339 UTC,
          "duration_seconds": int,
          "error_message":    string
        }
- [ ] Wire app config load into CLI startup; select NoOp or HTTP reporter
      based on `reporting.enabled`.
- [ ] Show reporting status (enabled/disabled) in CLI main menu header.
- [ ] Test against a local mock server before wiring to production.

Deliverables:
- [ ] `{RootDir}/config.toml` with reporting section
- [ ] `HTTPReporter` implementation
- [ ] "Configure Reporting" menu flow
- [ ] locked payload schema (v1)

Success looks like:
- After every run (manual or scheduled) the result is POSTed to the server.
- If the server is unreachable the backup job is unaffected.
- The node token never appears in logs or stdout.
