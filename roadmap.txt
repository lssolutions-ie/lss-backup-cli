LSS Backup CLI V2 Roadmap
=========================

Purpose
-------
This roadmap is for building a cleaner, safer, and more maintainable v2 of
LSS Backup CLI while preserving the strongest ideas from v1:

- each backup job should be independently restorable
- the job definition should be durable across reinstall/update
- import/export should remain first-class
- restic and rsync should both be supported
- the product should stay operator-friendly
- nodes should report backup status to a central management server


Guiding Principles
------------------
- One job definition file should be the source of truth.
- Runtime artifacts should be disposable and rebuildable.
- Jobs should remain operationally independent.
- Backup logic should live in shared code, not duplicated per job.
- Config should be treated as data, not executable script.
- Secrets should be stored separately from general config.
- Safety and recoverability matter more than cleverness.
- Reporting must never fail or block a backup job.

Milestone M5: Scheduling
------------------------
Status: COMPLETE

Note: Implemented via a long-running daemon process rather than OS-level
cron or Task Scheduler entries per job. This was a deliberate architectural
decision made during implementation — it gives more control and avoids
platform-specific scheduler fragmentation.

Tasks:
- [x] Store schedule in job config (manual / daily / weekly / monthly / cron).
- [x] Cron expression validation and English description.
- [x] Daemon process: load jobs, compute next run, sleep, run, reschedule.
- [x] Config reload every 60s (polling) + on-demand via signal file.
- [x] Graceful shutdown on SIGTERM / os.Interrupt.
- [x] Service installation: systemd (Linux), launchd (macOS), Task Scheduler (Windows).
- [x] Daemon log written to daemon.log on all platforms.
- [x] Daemon status shown in main menu and About screen.
- [x] Restart Daemon from Settings menu.

- [ ] Support failure-only vs always notification modes per job.


Milestone M7: Management Server Integration
-------------------------------------------
Status: NOT STARTED

Design decisions locked in:
- App-level config lives at `{RootDir}/config.toml`, mode 0600.
- Per-node token stored there; admin creates the token in the dashboard.
- Authorization via `Authorization: Bearer {token}` header.
- Payload always includes `payload_version: "1"` for forward compatibility.
- 15-second HTTP timeout; failure logged as warning only — never blocks backup.
- No retry queue. Fire-and-forget.
- Node name is user-configurable (not derived from hostname).

Tasks:
- [ ] Add `AppConfig` struct with a `[reporting]` section:
        enabled     bool
        server_url  string
        node_token  string
        node_name   string
- [ ] Add load/save for `{RootDir}/config.toml`.
- [ ] Add "Configure Management Console" to Settings menu (currently stubbed).
- [ ] Implement `HTTPReporter` in `internal/reporting/`:
        - POST to `{server_url}/api/v1/status`
        - Auth: `Authorization: Bearer {node_token}`
        - 15s timeout, log warning on failure
- [ ] Lock the JSON payload shape:
        {
          "payload_version": "1",
          "node_name":        string,
          "job_id":           string,
          "job_name":         string,
          "program":          string,
          "status":           "success" | "failure",
          "started_at":       RFC3339 UTC,
          "finished_at":      RFC3339 UTC,
          "duration_seconds": int,
          "error_message":    string
        }
- [ ] Wire app config into CLI startup; select NoOp or HTTP reporter.
- [ ] Show reporting status in CLI header or About.

Deliverables:
- [ ] `{RootDir}/config.toml` with reporting section
- [ ] `HTTPReporter` implementation
- [ ] "Configure Management Console" menu flow
- [ ] locked payload schema (v1)

Dependency: Refactor M6 Reporter interface first.
